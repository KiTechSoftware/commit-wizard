# 📗 External Contributor Guide

## Contributing Guide

Thanks for contributing!

This repository uses a structured workflow internally, but contributions are kept simple and flexible.

---

## ✅ What matters for your pull request

We focus on:

* Correct **target branch**
* Clear, focused changes
* Passing **CI checks**
* Proper **PR title (Conventional Commits)**
* Constructive review participation

Your branch name does **not** need to follow internal naming conventions.

---

## 🌿 Where to open your PR

* Most contributions → `develop`
* Bug fixes → `develop`

If a different target is needed, maintainers will guide you.

> ⚠️ PRs targeting the wrong branch will fail validation checks.

---

## 📝 Commit Messages (Important)

We use **Conventional Commits** to generate changelogs and releases.

Format your **PR title** like:

* `feat: add new feature`
* `fix: resolve issue with login`
* `refactor: simplify logic`
* `docs: update documentation`

> Your PR will be **squash merged**, so the PR title becomes the final commit message.

---

## 🚧 Work in Progress

Draft PRs are welcome.

Before merge:

* Changes must be clean and scoped
* CI must pass
* Review feedback must be addressed

---

## 🚀 Releases

Releases are handled by maintainers.

You do **not** need to:

* Create release branches
* Tag versions
* Manage versioning

---

## 📦 Summary

* Branch name → flexible
* PR target → must be correct
* PR title → must follow Conventional Commits
* CI → must pass
* Reviews → required

---

If you’re unsure about anything, open a PR — we’ll help guide you.
